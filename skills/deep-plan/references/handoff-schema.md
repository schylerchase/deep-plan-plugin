# Portable Handoff Bundle Schema

## Status

**Bundle version:** `"1.0"` (initial schema for deep-plan v1.2 cross-model handoff)
**Loaded by:** `/deep-plan:export-plan` during Phase 13 export; Phase 14 import tooling consumes the same contract.
**Purpose:** Define a single Markdown bundle format that preserves planning artifacts across models and plugins without losing bytes from the required source files.

## Versioning Rules

`bundle_version` is required and must be a quoted semver string. Version `"1.0"` is the initial portable handoff schema.

Minor version bumps are backward-compatible: a parser that understands `1.0` may read a later `1.x` bundle by accepting known fields and ignoring unknown optional fields. Major version bumps may break compatibility; import tooling must warn or stop when it sees an unsupported major version.

Exporters must emit the lowest schema version that accurately represents the bundle. For Phase 13, all emitted bundles use:

```yaml
bundle_version: "1.0"
```

## File Shape

A handoff bundle is one Markdown file:

1. YAML frontmatter delimited by `---`.
2. A Markdown body split by literal, line-anchored section markers.
3. Verbatim artifact content under each section marker.

Parsers split only on section markers that begin at the start of a line and are not inside a fenced code block. Whitespace before a marker means the line is content, not a marker. This prevents examples inside the embedded PLAN.md from terminating a bundle section early.

## Frontmatter Fields

| Field | Type | Required | Semantics |
|-------|------|----------|-----------|
| `bundle_version` | string | yes | Semver schema version. Initial value is `"1.0"`. |
| `source_model` | string | yes | Model identifier that created the bundle, such as `"claude-opus-4-7"` or `"gpt-5"`. |
| `source_plugin` | string | yes | Plugin and version that emitted the bundle, such as `"deep-plan@1.2.0"`. |
| `source_repo_id` | string | yes | Privacy-preserving repository identifier: `sha256(git_origin)[:12]`. |
| `exported_at` | string | yes | ISO-8601 UTC timestamp for export time. |
| `phase_id` | string | yes | Phase directory identifier, such as `"13-bundle-schema-export"`. |
| `phase_name` | string | yes | Human-readable phase name. |
| `target_model_hint` | enum | yes | One of `claude`, `codex`, or `generic`. Metadata only; content remains neutral. |
| `sections_included` | string array | yes | Exact set of body sections present. Subset of `PLAN`, `CONTEXT`, `RESEARCH`, `INTEL_SUMMARY`. |
| `original_paths` | map | yes | Source artifact paths relative to repo root. `plan` and `context` are required; `research` is optional. |

### original_paths Shape

```yaml
original_paths:
  plan: ".planning/phases/13-bundle-schema-export/13-01-PLAN.md"
  context: ".planning/phases/13-bundle-schema-export/13-CONTEXT.md"
  research: ".planning/phases/13-bundle-schema-export/13-RESEARCH.md"
```

`original_paths.plan` and `original_paths.context` are required because `PLAN` and `CONTEXT` are required sections. `original_paths.research` is present only when a `RESEARCH` section is included.

## Body Section Markers

Section markers are literal Markdown headings:

```markdown
## --- BUNDLE SECTION: PLAN ---
## --- BUNDLE SECTION: CONTEXT ---
## --- BUNDLE SECTION: RESEARCH ---
## --- BUNDLE SECTION: INTEL_SUMMARY ---
```

Required sections:

| Section | Required | Content rule |
|---------|----------|--------------|
| `PLAN` | yes | Verbatim copy of source `PLAN.md`, including its own frontmatter. |
| `CONTEXT` | yes | Verbatim copy of source `CONTEXT.md`. |
| `RESEARCH` | no | Verbatim copy of source `RESEARCH.md` when present and not suppressed. |
| `INTEL_SUMMARY` | no | Distilled planning intel generated by the algorithm in `skills/deep-plan/references/intel-distill.md`. |

`PLAN` and `CONTEXT` must be present and non-empty. `RESEARCH` and `INTEL_SUMMARY` are optional; `--minimal` exports omit both and list only `PLAN` and `CONTEXT` in `sections_included`.

The `PLAN` section may contain `routing.handoff_chain` frontmatter as documented in `skills/deep-plan/references/plan-template.md`. Exporters preserve that frontmatter verbatim inside the `PLAN` section.

## Example Bundle

```markdown
---
bundle_version: "1.0"
source_model: "gpt-5"
source_plugin: "deep-plan@1.2.0"
source_repo_id: "6b86b273ff34"
exported_at: "2026-04-30T16:20:00Z"
phase_id: "13-bundle-schema-export"
phase_name: "Bundle Schema + Export"
target_model_hint: "codex"
sections_included:
  - PLAN
  - CONTEXT
  - INTEL_SUMMARY
original_paths:
  plan: ".planning/phases/13-bundle-schema-export/13-01-PLAN.md"
  context: ".planning/phases/13-bundle-schema-export/13-CONTEXT.md"
---

## --- BUNDLE SECTION: PLAN ---
---
phase: 13-bundle-schema-export
plan: 01
type: execute
---

<objective>
Build the export side of cross-model planning handoff.
</objective>

## --- BUNDLE SECTION: CONTEXT ---
# Phase 13: Bundle Schema + Export - Context

<domain>
This phase introduces the export side of cross-model planning handoff.
</domain>

## --- BUNDLE SECTION: INTEL_SUMMARY ---
### Dependencies (top 20)

- No runtime dependency changes required for the schema document.

### Files (top 30 by size + 10 recent edits)

- `skills/deep-plan/references/handoff-schema.md`

### Public APIs

- `/deep-plan:export-plan`

### Architecture overview

The bundle is a Markdown file with YAML metadata and line-anchored section markers.

### Research summaries

- Phase 13 context establishes export-only scope; import is Phase 14.
```

## Validation Contract

Importers and validators must apply these checks before consuming section content:

| Check | Rule | Failure behavior |
|-------|------|------------------|
| Frontmatter parse | YAML frontmatter must parse as a mapping. | Reject the bundle as malformed frontmatter. |
| Required fields | All frontmatter fields listed above must be present. | Reject the bundle and report missing field names. |
| Version | `bundle_version` must be semver; `"1.0"` is supported. | Reject unsupported major versions; warn for newer minor versions. |
| Target hint | `target_model_hint` must be `claude`, `codex`, or `generic`. | Reject the bundle and report the invalid value. |
| Section set | `sections_included` must match actual section markers exactly. | Reject the bundle and report missing or extra sections. |
| Required sections | `PLAN` and `CONTEXT` markers must exist and their content must be non-empty. | Reject the bundle. |
| Optional sections | `RESEARCH` and `INTEL_SUMMARY` may be absent only when omitted from `sections_included`. | Accept absence; do not synthesize placeholder content. |
| Duplicate markers | If a marker appears more than once outside fenced code blocks, only the first marker wins and later duplicates are treated as content inside the winning section. | Warn about duplicate markers; preserve bytes as content. |
| Original paths | `original_paths.plan` and `original_paths.context` must be strings. | Reject the bundle. |

Malformed frontmatter is a hard error because section parsing depends on trusted metadata. Unknown frontmatter fields are allowed for forward compatibility and must be preserved when rewriting a bundle.

## Optional-Section Handling

Optional sections are controlled by `sections_included`, not by guessing from file names. A parser must not require `RESEARCH` or `INTEL_SUMMARY` when they are absent from `sections_included`.

When `RESEARCH` is present, it is copied verbatim from the source research artifact. When `INTEL_SUMMARY` is present, its content follows `skills/deep-plan/references/intel-distill.md`; importers should treat it as advisory context, not as a replacement for `PLAN` or `CONTEXT`.

## Round-Trip Guarantee

The bundle is lossless for required planning artifacts. Exporters must copy `PLAN` and `CONTEXT` bytes verbatim into their sections, including frontmatter, blank lines, indentation, and trailing newlines. Importers must write those section bodies back byte-for-byte.

Round-trip guarantee:

```text
source PLAN.md bytes == imported PLAN.md bytes
source CONTEXT.md bytes == imported CONTEXT.md bytes
```

Optional `RESEARCH` receives the same verbatim guarantee when included. `INTEL_SUMMARY` has no verbatim source artifact because it is generated from the distillation algorithm.
